HISTORY & CORE PRINCIPLES

From Caesar's cipher to quantum-resistant algorithms — the evolution of secret communication, and the three pillars that govern all modern cryptography.

CIA triadKerckhoffs's principle confusion & diffusionperfect secrecy

THE THREE GOALS OF CRYPTOGRAPHY

core

CONFIDENTIALITY

Only authorised parties can read the data. Achieved via encryption. Even if intercepted, the ciphertext is unreadable without the key.

→ AES, RSA, TLS

INTEGRITY

Data has not been tampered with in transit. Achieved via hashing + MACs. Any modification changes the hash/MAC — detectable.

→ SHA-256, HMAC

AUTHENTICITY

Proves who sent the message. Achieved via digital signatures + PKI. Only the private key holder could have signed it.

→ RSA-sig, ECDSA, X.509

KERCKHOFFS'S PRINCIPLE & SECURITY BY OBSCURITY

philosophy
▾

Kerckhoffs's Principle (1883): A cryptosystem should be secure even if everything about the system, except the key, is public knowledge.

Why This Matters
Modern algorithms like AES are fully public. Their source code, mathematics, and structure are all published. Security comes entirely from the key. This is the opposite of "security by obscurity" — which fails because secrets leak.
ConceptDefinitionExample
ConfusionEach bit of ciphertext depends on many parts of the key (substitution)S-boxes in AES
DiffusionChanging one plaintext bit changes ~50% of ciphertext bits (permutation)ShiftRows in AES
Avalanche EffectSmall change in input → drastically different outputSHA-256, AES
Perfect SecrecyCiphertext reveals zero information about plaintext — even with infinite computeOne-Time Pad (OTP)

CAESAR CIPHER — Shift Cipher

python
def caesar(text, shift):
    result = ""
    for ch in text:
        if ch.isalpha():
            base = ord('A') if ch.isupper() else ord('a')
            result += chr(((ord(ch) - base + shift) % 26) + base)
        else:
            result += ch
    return result

# Encrypt
print(caesar("Hello World", 3))   # → Khoor Zruog

# Brute force all 25 shifts
ciphertext = "Khoor Zruog"
for shift in range(1, 26):
    print(f"Shift {shift:2d}: {caesar(ciphertext, -shift)}")

Vigenère — uses a keyword to vary the shift. Each letter of the key shifts the corresponding plaintext letter. Vulnerable to Kasiski analysis.

python
def vigenere(text, key, encrypt=True):
    result, ki = "", 0
    key = key.upper()
    for ch in text.upper():
        if ch.isalpha():
            shift = ord(key[ki % len(key)]) - 65
            if not encrypt: shift = -shift
            result += chr(((ord(ch) - 65 + shift) % 26) + 65)
            ki += 1
        else: result += ch
    return result

print(vigenere("ATTACKATDAWN", "LEMON"))
# → LXFOPVEFRNHR
Frequency Analysis
Classical ciphers are defeated by frequency analysis: in English, 'E' appears ~12.7% of the time. If ciphertext letter 'X' appears ~12.7%, X likely maps to E. This is why modern ciphers must produce statistically uniform output.

SYMMETRIC ENCRYPTION

Same key encrypts and decrypts. Extremely fast — used for bulk data. The challenge: key distribution.

SYMMETRIC ENCRYPTION MODEL SENDER SHARED KEY RECEIVER ┌──────────┐ ┌───────────┐ ┌──────────┐ │ PLAINTEXT│──encrypt───► │ AES-256 │ ──decrypt──► │ PLAINTEXT│ └──────────┘ └───────────┘ └──────────┘ │ CIPHERTEXT (public channel) looks like: 3a7f9b2c4e8d1a6f... Problem: How do sender and receiver securely share the key? Answer: Use asymmetric crypto (RSA/ECDH) just for key exchange, then use symmetric crypto (AES) for actual data. This is exactly what TLS does.

AES — Advanced Encryption Standard

industry standard

AES is the gold standard symmetric cipher. Block cipher operating on 128-bit blocks, with key sizes of 128, 192, or 256 bits. Adopted by NIST in 2001, used in TLS, disk encryption, WiFi (WPA2), and more.

import os

# Generate a random 256-bit key
key = os.urandom(32)        # 32 bytes = 256 bits
nonce = os.urandom(12)      # 12 bytes for GCM nonce

print(f"Decrypted:  {recovered.decode()}")

ECB Mode — The Penguin Problem
ECB encrypts each 128-bit block independently. Identical plaintext blocks produce identical ciphertext blocks. Encrypting a bitmap image with ECB preserves patterns — the structure of the image is visible in the ciphertext. Never use ECB. Always use GCM or CTR.

Stream Ciphers & XOR — The Building Block

theory
▾

Stream ciphers encrypt one bit/byte at a time by XORing plaintext with a pseudorandom keystream. CTR mode turns AES into a stream cipher.

python — XOR cipher fundamentals
# XOR Properties:
# A XOR B = C    (encrypt)
# C XOR B = A    (decrypt — same operation!)
# A XOR A = 0    (XOR with itself = zero)
# A XOR 0 = A    (XOR with zero = unchanged)

def xor_cipher(data: bytes, key: bytes) -> bytes:
    return bytes(a ^ b for a, b in zip(data, (key * (len(data)//len(key)+1))[:len(data)]))

msg = b"ATTACK AT DAWN"
key = b"SECRET"
ct  = xor_cipher(msg, key)
pt  = xor_cipher(ct, key)   # same function = decryption
print(pt)  # b'ATTACK AT DAWN'

# CTF: if two messages encrypted with SAME key:
# CT1 XOR CT2 = PT1 XOR PT2  (key cancels out!)
# This is the "two-time pad" vulnerability
ct1 = xor_cipher(b"Hello World!!", key)
ct2 = xor_cipher(b"Secret Msg!!!", key)
combined = bytes(a ^ b for a,b in zip(ct1, ct2))
print(combined.hex())  # leaks XOR of both plaintexts!

ASYMMETRIC ENCRYPTION

Two mathematically linked keys — one public, one private. Solves the key distribution problem. Slower than symmetric, so used only for key exchange and signatures.

ASYMMETRIC ENCRYPTION — Two Keys, One Pair Alice generates key pair: Public Key → shared freely with anyone (lock anyone can close) Private Key → NEVER shared, kept secret (key only Alice has) Encryption (Bob → Alice): Bob uses Alice's PUBLIC key to encrypt → only Alice's private key can decrypt Digital Signature (Alice → Bob): Alice uses her PRIVATE key to sign → anyone with public key can verify Mathematics: easy to compute, computationally infeasible to reverse RSA: factoring large primes (n = p × q, hard to reverse) ECC: elliptic curve discrete logarithm problem DH/ECDH: discrete logarithm problem

RSA — Rivest–Shamir–Adleman

most studied

RSA security is based on the difficulty of factoring the product of two large primes. Key generation: pick two large primes p, q → n = p×q → compute totient φ(n) = (p-1)(q-1) → choose e (public exponent, usually 65537) → find d such that e×d ≡ 1 mod φ(n).

python — RSA from scratch (small)
# Small RSA demo (not secure — for learning only)
p, q = 61, 53               # real: 2048+ bit primes
n = p * q                    # n = 3233
phi = (p - 1) * (q - 1)   # φ(n) = 3120
e = 17                       # public exponent (gcd(e,φ)=1)

# Extended Euclidean to find d
def modinv(a, m):
    g, x = extended_gcd(a % m, m)
    return x % m
def extended_gcd(a, b):
    if a == 0: return b, 0
    g, x = extended_gcd(b % a, a)
    return g, 1 - (b // a) * x

d = modinv(e, phi)           # d = 2753 (private key)

# Encrypt:  c = m^e mod n
m = 65                       # message = 65
c = pow(m, e, n)             # ciphertext = 2790
print(f"Encrypted: {c}")

# Decrypt:  m = c^d mod n
print(f"Decrypted: {pow(c, d, n)}")  # → 65
python — RSA with library
from cryptography.hazmat.primitives.asymmetric import rsa, padding
from cryptography.hazmat.primitives import hashes

# Generate 2048-bit RSA key pair
private_key = rsa.generate_private_key(
    public_exponent=65537,
    key_size=2048
)
public_key = private_key.public_key()

# Encrypt with public key (OAEP padding)
ct = public_key.encrypt(
    b"Top secret data",
    padding.OAEP(
        mgf=padding.MGF1(algorithm=hashes.SHA256()),
        algorithm=hashes.SHA256(),
        label=None
    )
)

# Decrypt with private key
pt = private_key.decrypt(
    ct,
    padding.OAEP(
        mgf=padding.MGF1(algorithm=hashes.SHA256()),
        algorithm=hashes.SHA256(),
        label=None
    )
)
print(pt)  # b'Top secret data'
RSA Key Sizes
1024-bit: Broken — do not use.  2048-bit: Current minimum.  4096-bit: Recommended for long-term security. RSA is ~1000× slower than AES, which is why it's only used for key exchange and signatures, not bulk encryption.

python — ECDH key exchange
from cryptography.hazmat.primitives.asymmetric.x25519 import X25519PrivateKey

# Alice generates her keys
alice_private = X25519PrivateKey.generate()
alice_public  = alice_private.public_key()

# Bob generates his keys
bob_private = X25519PrivateKey.generate()
bob_public  = bob_private.public_key()

# Key exchange — both derive the SAME shared secret
alice_shared = alice_private.exchange(bob_public)
bob_shared   = bob_private.exchange(alice_public)

print(alice_shared == bob_shared)  # True!
print(alice_shared.hex())          # same 32-byte secret
# This shared secret can now derive an AES key (HKDF)

# Hash a file (file integrity check)
def hash_file(path):
    h = hashlib.sha256()
    with open(path, 'rb') as f:
        for chunk in iter(lambda: f.read(4096), b''):
            h.update(chunk)
    return h.hexdigest()

print(hash_file("/etc/passwd"))

SHAttered Attack (2017)
Google demonstrated the first SHA-1 collision: two different PDF files with the same SHA-1 hash. This took ~9.2 quintillion computations. SHA-1 is now officially broken for security purposes. All modern systems use SHA-256 or SHA-3.

DIGITAL SIGNATURES

Prove authenticity and integrity simultaneously. Only the private key holder can sign; anyone with the public key can verify.

DIGITAL SIGNATURE — SIGN & VERIFY SIGNING (Alice → Bob): 1. Alice computes: hash = SHA-256(document) 2. Alice encrypts: signature = RSA_encrypt(hash, Alice's_PRIVATE_key) 3. Alice sends: document + signature VERIFICATION (Bob verifies): 1. Bob computes: hash1 = SHA-256(received_document) 2. Bob decrypts: hash2 = RSA_decrypt(signature, Alice's_PUBLIC_key) 3. Bob checks: hash1 == hash2 → VALID (unmodified, from Alice) hash1 != hash2 → INVALID (tampered or wrong signer) Non-repudiation: Alice cannot deny signing — only her private key could have produced it.
python — RSA & ECDSA signatures
from cryptography.hazmat.primitives.asymmetric import rsa, ec, padding
from cryptography.hazmat.primitives import hashes
from cryptography.hazmat.primitives.asymmetric.utils import decode_dss_signature

# ─── RSA Signature ───────────────────────────────
priv = rsa.generate_private_key(public_exponent=65537, key_size=2048)
pub  = priv.public_key()

document = b"I agree to transfer $1000 to Bob"
signature = priv.sign(document, padding.PKCS1v15(), hashes.SHA256())

# Verify (raises InvalidSignature if tampered)
try:
    pub.verify(signature, document, padding.PKCS1v15(), hashes.SHA256())
    print("RSA Signature: VALID ✓")
except Exception:
    print("RSA Signature: INVALID ✗")

# ─── ECDSA Signature (smaller, faster) ───────────
ec_priv = ec.generate_private_key(ec.SECP256R1())  # P-256
ec_pub  = ec_priv.public_key()

ec_sig = ec_priv.sign(document, ec.ECDSA(hashes.SHA256()))
try:
    ec_pub.verify(ec_sig, document, ec.ECDSA(hashes.SHA256()))
    print("ECDSA Signature: VALID ✓")
except:
    print("ECDSA Signature: INVALID ✗")

KEY EXCHANGE — DIFFIE-HELLMAN

The elegant solution to the key distribution problem: two parties derive the same shared secret over a public channel without ever transmitting it.

ALICE

g=5, p=23
private (secret)
a = 6
computes public value
A = g^a mod p
A = 5^6 mod 23 = 8
shared secret
s = B^a mod p = 2^6 mod 23 = 2
← A = 8
sent to Bob
B = 2 →
sent to Alice
PUBLIC
CHANNEL
Eve sees:
g,p,A,B
NOT s!

BOB

public params
g=5, p=23
private (secret)
b = 15
computes public value
B = g^b mod p
B = 5^15 mod 23 = 2
shared secret
s = A^b mod p = 8^15 mod 23 = 2

Both independently computed s=2. Eve cannot derive s from g,p,A,B — this is the discrete log problem.

Perfect Forward Secrecy (PFS)
Ephemeral DH (DHE/ECDHE) generates a fresh DH key pair for every session. Even if the server's long-term private key is later compromised, past sessions cannot be decrypted because the ephemeral keys were discarded. TLS 1.3 requires ECDHE — PFS is mandatory.

MAC & HMAC

Message Authentication Codes combine integrity with authentication — proving data wasn't tampered with AND came from someone holding the secret key.

MAC vs Hash

A hash verifies integrity but NOT authenticity — anyone can recompute a hash and replace it. A MAC requires a secret key, so only the keyholder can produce a valid MAC.

MAC(key, message) → tag ───────────────────────── Verify: recompute MAC, check if tags match. Attacker can't forge without the key.

HMAC Construction

HMAC (Hash-based MAC) uses a hash function internally with the key applied twice for security.

HMAC(K, M) = H( (K ⊕ opad) ‖ H( (K ⊕ ipad) ‖ M ) ) opad = 0x5c5c5c... ipad = 0x363636... Secure even if underlying hash has length extension vulnerabilities.
python — HMAC implementation
import hmac, hashlib, os, secrets

key = os.urandom(32)        # 256-bit secret key
message = b"Transfer $100 to Alice"

# Compute HMAC-SHA256
mac = hmac.new(key, message, hashlib.sha256).hexdigest()
print(f"HMAC: {mac}")

# Verify (use hmac.compare_digest — timing-safe!)
received_mac = hmac.new(key, message, hashlib.sha256).digest()
computed_mac = hmac.new(key, message, hashlib.sha256).digest()
if hmac.compare_digest(received_mac, computed_mac):
    print("Authentic ✓")

# WRONG: don't use ==  for MAC comparison → timing attack!
# if received_mac == computed_mac:  ← VULNERABLE
# ALWAYS use hmac.compare_digest() for constant-time comparison

# Using cryptography library (CMAC over AES)
from cryptography.hazmat.primitives.cmac import CMAC
from cryptography.hazmat.primitives.ciphers import algorithms

aes_key = os.urandom(16)
c = CMAC(algorithms.AES(aes_key))
c.update(b"data to authenticate")
print(c.finalize().hex())

PKI & X.509 CERTIFICATES

Public Key Infrastructure — the trust hierarchy that makes HTTPS work. Who vouches for whose public key?

CERTIFICATE CHAIN OF TRUST ROOT CA (Root Certificate Authority) └── Self-signed certificate, stored in your OS/browser └── Extremely carefully guarded (offline, air-gapped) └── Examples: DigiCert Root G1, Comodo, Let's Encrypt ISRG Root X1 INTERMEDIATE CA └── Certificate signed by Root CA └── Issues end-entity certificates day-to-day └── If compromised → revoked without touching Root END-ENTITY CERT (leaf) └── Certificate for google.com, github.com, etc. └── Contains: domain, public key, validity period, CA signature └── Verified by: browser checks CA signature chain up to trusted Root Browser verifies: 1. Cert not expired 2. Domain matches cert (CN or SAN) 3. Signature chain → trusted Root CA 4. Not revoked (CRL / OCSP check)

PASSWORD HASHING

Storing passwords correctly — why plain SHA-256 is WRONG and what to use instead.

Never Store Plaintext Passwords
Never store plaintext or simple hashes (MD5/SHA-256) of passwords. A leaked database of SHA-256 hashes can be cracked with a GPU at billions of guesses per second using rainbow tables. Always use purpose-built slow hash functions.

python — password hashing properly
import bcrypt
from cryptography.hazmat.primitives.kdf.scrypt import Scrypt
from cryptography.hazmat.primitives.kdf.argon2 import Argon2id  # cryptography>=41
import os

# ─── bcrypt ────────────────────────────────────────────────
password = b"my_secret_password"
hash_ = bcrypt.hashpw(password, bcrypt.gensalt(rounds=12))  # cost=12
print(hash_)  # $2b$12$SALT....HASH

# Verify
if bcrypt.checkpw(password, hash_):
    print("Password correct ✓")

# ─── scrypt ────────────────────────────────────────────────
salt = os.urandom(16)
kdf = Scrypt(salt=salt, length=32, n=2**14, r=8, p=1)
derived = kdf.derive(b"my_secret_password")
print(derived.hex())

# ─── Why salting matters ───────────────────────────────────
# Without salt: hash("password") is ALWAYS the same
# Attacker pre-computes: dict → hash lookup (rainbow table)
# With salt: hash("password" + random_salt) is unique each time
# bcrypt/scrypt/argon2 handle salting automatically — always use them

# ─── PBKDF2 (older but still acceptable) ─────────────────
import hashlib
dk = hashlib.pbkdf2_hmac('sha256', b'password', salt, 310000)
# 310,000 iterations = NIST minimum 2023 recommendation

HYBRID ENCRYPTION — How Real Systems Work

real-world pattern

TLS, PGP, and Signal all use hybrid encryption: asymmetric crypto to exchange a symmetric key, then symmetric crypto for the actual data. Best of both worlds.

python — complete hybrid encryption system
from cryptography.hazmat.primitives.asymmetric import rsa, padding
from cryptography.hazmat.primitives.ciphers.aead import AESGCM
from cryptography.hazmat.primitives import hashes, serialization
import os, json, base64

# ── KEY GENERATION ────────────────────────────────
def generate_rsa_keypair():
    priv = rsa.generate_private_key(public_exponent=65537, key_size=2048)
    return priv, priv.public_key()

# ── ENCRYPT (Sender) ──────────────────────────────
def encrypt(plaintext: bytes, recipient_pub_key) -> dict:
    # 1. Generate random AES-256 key
    aes_key = os.urandom(32)
    nonce   = os.urandom(12)

    # 2. Encrypt data with AES-GCM
    ciphertext = AESGCM(aes_key).encrypt(nonce, plaintext, None)

    # 3. Encrypt AES key with recipient's RSA public key
    enc_key = recipient_pub_key.encrypt(
        aes_key,
        padding.OAEP(padding.MGF1(hashes.SHA256()), hashes.SHA256(), None)
    )
    return {
        "enc_key":    base64.b64encode(enc_key).decode(),
        "nonce":      base64.b64encode(nonce).decode(),
        "ciphertext": base64.b64encode(ciphertext).decode()
    }

# ── DECRYPT (Recipient) ───────────────────────────
def decrypt(package: dict, priv_key) -> bytes:
    enc_key    = base64.b64decode(package["enc_key"])
    nonce      = base64.b64decode(package["nonce"])
    ciphertext = base64.b64decode(package["ciphertext"])

    # 1. Decrypt AES key with RSA private key
    aes_key = priv_key.decrypt(
        enc_key,
        padding.OAEP(padding.MGF1(hashes.SHA256()), hashes.SHA256(), None)
    )
    # 2. Decrypt data with AES-GCM
    return AESGCM(aes_key).decrypt(nonce, ciphertext, None)

# If two RSA moduli share a prime factor: gcd(n1, n2) = p

def id_hash(h):
    hmap = {32: "MD5", 40: "SHA-1", 64: "SHA-256", 96: "SHA-384", 128: "SHA-512"}
    return hmap.get(len(h), "Unknown")
