OSI & TCP/IP Models

The conceptual frameworks that describe how data travels across networks — from application logic down to electrical signals.

The OSI (Open Systems Interconnection) model divides network communication into 7 abstraction layers. Each layer only communicates with the layers directly above and below it. This separation makes it easier to troubleshoot, design, and understand networks.

Memory Aid
Layer 7→1: "All People Seem To Need Data Processing" — Application, Presentation, Session, Transport, Network, Data Link, Physical

HTTP, FTP, DNS, SMTP, SSH
6
Presentation
TLS/SSL, JPEG, ASCII, encryption
5
Session
Session setup/teardown, NetBIOS
4

TCP (reliable), UDP (fast), ports
3
Network
IP addressing, routing, ICMP
2
Data Link
MAC addresses, Ethernet, ARP, switches
1

Cables, Wi-Fi, hubs, bits as signals

Encapsulation — How data travels DOWN the stack

Key Concept

As data moves down from Layer 7 to Layer 1, each layer wraps the data with its own header (and sometimes a trailer). This is called encapsulation. At the receiving end, each layer strips its header going back up.

[Layer 7] DATA [Layer 4] TCP HEADER | DATA → Segment [Layer 3] IP HEADER | TCP HEADER | DATA → Packet [Layer 2] ETH HEADER | IP HEADER | TCP HEADER | DATA | ETH TRAILER → Frame [Layer 1] 01001011 00101011 10110010... → Bits
Security Note
Wireshark shows you exactly this structure. When you capture a packet, you can expand each layer and see the raw headers — this is fundamental to understanding exploits that manipulate headers at specific layers (e.g., IP spoofing at Layer 3, ARP poisoning at Layer 2).

TCP/IP Model — The Real-World Simplification

Practical

The TCP/IP model (also called the Internet Model) collapses the 7 OSI layers into 4. This is what actually runs the internet.

TCP/IP LayerOSI Equivalent
ApplicationLayers 5, 6, 7
TransportLayer 4
InternetLayer 3
Network AccessLayers 1, 2

PDU Names by Layer

Layer 7 → Data
Layer 4 → Segment (TCP) / Datagram (UDP)
Layer 3 → Packet
Layer 2 → Frame
Layer 1 → Bits

IP Addressing & Subnetting

How devices are addressed on networks, and how networks are divided into manageable sub-networks.

IPv4 Structure

32-bit

IPv4 addresses are 32 bits, written as 4 octets in dotted-decimal notation.

192 . 168 . 1 . 105 11000000.10101000.00000001.01101001 8 bits 8 bits 8 bits 8 bits ←─────────── 32 bits total ──────────→
  • Each octet: 0–255
  • Total addresses: 2³² = ~4.3 billion
  • Nearly exhausted → hence IPv6

TCP vs UDP

The two transport-layer protocols that everything runs on — one guarantees delivery, one doesn't bother.

TCP — Transmission Control Protocol

Connection-oriented. Establishes a session before data flows. Guarantees:

  • ✓ Delivery (retransmission if lost)
  • ✓ Order (reassembled in sequence)
  • ✓ Error checking
  • ✗ Slower (more overhead)

Used by: HTTP, HTTPS, SSH, FTP, SMTP

UDP — User Datagram Protocol

Connectionless. Fire-and-forget. No handshake, no ACKs. Offers:

  • ✓ Very fast (low overhead)
  • ✓ Low latency
  • ✗ No delivery guarantee
  • ✗ No ordering

Used by: DNS, DHCP, VoIP, video streaming, gaming

TCP 3-Way Handshake — Connection Setup

Critical

Before any TCP data is exchanged, both sides synchronize sequence numbers via a 3-step handshake. This is one of the most important concepts in networking and security.

CLIENT SERVER SYN (seq=1000) ──────────────► Server receives SYN ◄────────────── SYN-ACK (seq=5000, ack=1001) Client receives SYN-ACK ACK (seq=1001, ack=5001) ─────────────► Connection ESTABLISHED ◄─────────────► DATA FLOWS ◄──────────────► TCP 4-way TEARDOWN (closing): FIN → FIN-ACK → FIN → FIN-ACK
Security — SYN Flood Attack
Attackers send thousands of SYN packets but never complete the handshake. The server allocates resources for each half-open connection until it runs out of memory — a classic DoS attack. Mitigated by SYN cookies.
bash — observe TCP handshake
# Capture TCP handshake on port 80 with tcpdump
sudo tcpdump -i eth0 'tcp port 80 and (tcp-syn!=0 or tcp-ack!=0)' -n

# See TCP flags in Wireshark filter
tcp.flags.syn==1 and tcp.flags.ack==0   # SYN packets only
tcp.flags.syn==1 and tcp.flags.ack==1   # SYN-ACK packets

# Check open TCP connections
ss -tn state established
netstat -tn | grep ESTABLISHED

TCP Flags Explained

HTTP & HTTPS

The protocol of the web. Every web request, API call, and web attack runs over HTTP. Understanding this at the raw level is fundamental to web security.

HTTP Request & Response Structure

Raw Protocol

HTTP Request:

HTTP/1.1 200 OK Content-Type: text/html Content-Length: 4523 Set-Cookie: session=xyz Server: Apache/2.4.38 X-Frame-Options: DENY ← blank line <html>...response body...</html>

HTTPS — TLS Handshake

Encryption

HTTPS = HTTP + TLS (Transport Layer Security). TLS encrypts the HTTP traffic so that a man-in-the-middle sees only gibberish.

CLIENT SERVER │ │ │── ClientHello (TLS version, ciphers) ──► │ │ │ │◄── ServerHello (chosen cipher) ─────── │ │◄── Certificate (public key + cert) ──── │ │◄── ServerHelloDone ─────────────────── │ │ │ │ [client verifies cert with CA chain] │ │── ClientKeyExchange ────────────────── │ (sends premaster secret) │── ChangeCipherSpec ─────────────────── │ │── Finished (encrypted) ────────────── │ │ │ │◄── ChangeCipherSpec ───────────────── │ │◄── Finished (encrypted) ────────────── │ │ │ │◄══════════ Encrypted HTTP data ══════► │
bash — inspect TLS certificates
# View TLS certificate details
echo | openssl s_client -connect google.com:443 2>/dev/null | openssl x509 -text -noout

# Check cipher suites and TLS version
openssl s_client -connect target.com:443 -tls1_2

# Test with curl — follow redirects, show headers
curl -Lv https://target.com 2>&1 | grep -E "TLS|SSL|cert"

# Check HTTP security headers
curl -sI https://target.com | grep -iE "strict|content-security|x-frame|x-content"

Routing & Switching

How packets find their way across networks, and how devices within a network communicate via MAC addresses.

Switches (Layer 2)

Switches forward frames based on MAC addresses. They maintain a MAC address table (CAM table) mapping MAC → port.

Switch CAM Table: MAC Address Port AA:BB:CC:11:22:33 1 DD:EE:FF:44:55:66 2 11:22:33:44:55:66 3 Frame arrives on Port 1 for DD:EE... → Switch looks up MAC in table → Forwards only to Port 2 (not broadcast)
Attack — MAC Flooding
Sending thousands of fake MACs fills the CAM table. Switch enters "fail-open" mode and broadcasts everything → attacker can sniff all traffic (like a hub).

Routers (Layer 3)

Routers forward packets between different networks using IP addresses and a routing table.

Routing Table: Destination Gateway Interface 192.168.1.0/24 0.0.0.0 eth0 (direct) 10.0.0.0/8 192.168.1.1 eth0 0.0.0.0/0 192.168.1.1 eth0 ← default route Packet for 8.8.8.8: → No specific match → Use default route → forward to 192.168.1.1
bash
# View routing table
ip route show
route -n         # older syntax

# Trace path to destination
traceroute 8.8.8.8
tracepath google.com

ARP — Address Resolution Protocol

Layer 2/3 Bridge

ARP resolves IP addresses to MAC addresses within a local subnet. When a device wants to send a packet to 192.168.1.5, it broadcasts: "Who has 192.168.1.5? Tell 192.168.1.10"

ARP Request (Broadcast): Who has 192.168.1.5? Tell 192.168.1.10 Source MAC: AA:BB:CC:11:22:33 Destination MAC: FF:FF:FF:FF:FF:FF ← broadcast ARP Reply (Unicast): 192.168.1.5 is at DD:EE:FF:44:55:66 ARP Cache (shows who's on local network): 192.168.1.1 gateway_mac eth0 192.168.1.5 DD:EE:FF:... eth0
bash — ARP commands
# View ARP cache
arp -n
ip neigh show

# ARP scan the local subnet (host discovery)
arp-scan --localnet
nmap -sn 192.168.1.0/24  # ping sweep (uses ARP on LAN)
Attack — ARP Poisoning / Spoofing
Attacker sends fake ARP replies: "192.168.1.1 is at attacker's MAC". Victims update their ARP cache. All traffic meant for the gateway now goes to the attacker → Man-in-the-Middle. Tool: arpspoof or ettercap.

NAT — Network Address Translation

How the Internet Works

NAT allows multiple devices with private IPs to share a single public IP. Your home router does this.

Private Network Router/NAT Public Internet 192.168.1.10:54321 ──► 203.0.113.5:40001 ──► 8.8.8.8:53 (DNS) 192.168.1.11:54322 ──► 203.0.113.5:40002 ──► 1.1.1.1:443 (HTTPS) 192.168.1.12:54323 ──► 203.0.113.5:40003 ──► 8.8.8.8:53 (DNS) NAT Translation Table: Internal External 192.168.1.10:54321 ↔ 203.0.113.5:40001 192.168.1.11:54322 ↔ 203.0.113.5:40002
Security Implication
NAT provides implicit protection — inbound connections are blocked unless explicitly configured (port forwarding). Attackers can't directly reach 192.168.1.x devices from the internet. This is why reverse shells are used: the victim connects OUT, bypassing NAT and firewalls.

wireshark security filters
# Find cleartext passwords in HTTP POST
http.request.method == "POST"

# TCP SYN scan detection (many SYN, no SYN-ACK)
tcp.flags.syn==1 and tcp.flags.ack==0

# Large number of DNS queries (DNS tunneling?)

http.response.code >= 400

# Combine filters with AND / OR
ip.src==192.168.1.5 and tcp.dstport==80

Nmap Output — Reading Results

Analysis
