SOC Structure & Roles

The Security Operations Center is the nerve center of defensive security — monitoring, detecting, analyzing, and responding to threats 24/7.

SOC ORGANIZATIONAL STRUCTURE TIER 1 — Alert Triage Analysts ├── Monitor SIEM dashboards, alert queues ├── Initial triage: true positive vs false positive ├── Escalate confirmed incidents to Tier 2 └── Response time: seconds to minutes TIER 2 — Incident Responders ├── Deep investigation of escalated alerts ├── Threat containment and eradication ├── Correlation across multiple data sources └── Coordinate with IT for remediation TIER 3 — Threat Hunters / Senior Analysts ├── Proactive threat hunting (no alert needed) ├── Malware analysis and reverse engineering ├── Develop detection rules and playbooks └── Threat intelligence integration SOC Manager → CISO → Executive Leadership Metrics, SLAs, tool procurement, team training

grep "Failed password" /var/log/auth.log | \

# Detect scanners/attack tools
grep -iE "sqlmap|nikto|nmap|masscan|gobuster" /var/log/nginx/access.log

Get-WinEvent -FilterHashtable @{
  LogName='Security'; Id=4625
} -MaxEvents 100 | Select-Object TimeCreated,
  @{N='User';E={$_.Properties[5].Value}},

# ── PROCESS CREATION (4688) ───────────────────
Get-WinEvent -FilterHashtable @{
  LogName='Security'; Id=4688

  LogName='System'; Id=7045

  @{N='Service';E={$_.Properties[0].Value}},

  LogName='Security'; Id=1102

SIEM — Splunk & ELK Stack

Security Information and Event Management — aggregates logs from across the infrastructure, correlates events, and generates alerts from rules and ML detections.

Splunk — Search Processing Language (SPL)

SplunkSPL
SPL QUERY STRUCTURE index=windows EventCode=4625 | stats count by src_ip | sort -count | head 10 ↑ Search ↑ Filter ↑ Aggregation ↑ Sort ↑ Limit Data flow: Search → Filter → Eval → Stats → Sort → Table → Visualize
Splunk SPL queries for SOC analysts
SPL
| ── BRUTE FORCE DETECTION ──────────────────────────────────────────────
index=windows EventCode=4625
| stats count by src_ip, user, dest_host
| where count > 10
| sort -count
| eval severity=if(count>50,"CRITICAL",if(count>20,"HIGH","MEDIUM"))

| ── SUCCESSFUL LOGIN AFTER FAILURES (Account Compromise) ──────────────
index=windows EventCode IN (4625,4624)
| eval event_type=if(EventCode==4624,"success","failure")
| stats values(event_type) as events, count by src_ip, user
| where mvfind(events,"failure")>=0 AND mvfind(events,"success")>=0

| ── SUSPICIOUS PROCESS EXECUTION ──────────────────────────────────────
index=windows EventCode=4688
| where match(New_Process_Name, "(?i)powershell|cmd|wscript|cscript|mshta|rundll32")
| table _time, host, Account_Name, New_Process_Name, Process_Command_Line
| sort -_time

| ── LATERAL MOVEMENT DETECTION ─────────────────────────────────────────
index=windows EventCode=4624 Logon_Type=3
| stats dc(dest_host) as unique_dests, values(dest_host) as destinations by src_ip, user
| where unique_dests > 5
| sort -unique_dests

| ── DATA EXFILTRATION DETECTION ────────────────────────────────────────
index=firewall action="allowed" direction="outbound"
| stats sum(bytes_out) as total_bytes by src_ip, dest_ip, dest_port
| where total_bytes > 100000000
| eval total_MB=round(total_bytes/1024/1024,2)
| sort -total_MB

| ── DNS TUNNELING DETECTION ────────────────────────────────────────────
index=dns query_type="TXT"
| eval query_len=len(query)
| where query_len > 60
| stats count, values(query) as queries by src_ip, dest_domain
| sort -count

| ── LOG CLEARED ALERT ──────────────────────────────────────────────────
index=windows EventCode=1102 OR EventCode=104
| table _time, host, Account_Name, EventCode
| eval severity="CRITICAL — Possible Cover-Up"

Network Monitoring & Analysis

Capturing, analyzing, and alerting on network traffic. IDS/IPS, packet analysis, netflow monitoring, and baseline deviation detection.

Wireshark / tcpdump — packet analysis
bash
# ── TCPDUMP CAPTURE ───────────────────────────
# Capture all traffic, save to file
sudo tcpdump -i eth0 -w capture.pcap

# Capture specific host
sudo tcpdump -i eth0 host 192.168.1.5 -w host.pcap

# Capture specific port
sudo tcpdump -i eth0 port 443 -w https.pcap

# Detect port scanning (many SYNs, no SYN-ACK)
sudo tcpdump -i eth0 'tcp[tcpflags] == tcp-syn'

# ── WIRESHARK DISPLAY FILTERS ─────────────────
# Detect SYN scans
tcp.flags.syn==1 and tcp.flags.ack==0

# ARP poisoning (multiple IPs, same MAC)
arp.duplicate-address-detected

# DNS queries with long names (tunneling)
dns.qry.name.len > 50

# Large outbound transfers
ip.src == 192.168.1.0/24 and tcp.len > 5000

# Suspicious cleartext credentials
http.request.method == "POST" and
http contains "password"

# Beaconing — regular interval connections
frame contains "User-Agent: " and
http.request.uri contains "/beacon"
Zeek / Suricata / Snort
bash/rules
# ── ZEEK (network analysis framework) ─────────
# Zeek auto-generates structured logs:
# conn.log — all connections
# dns.log  — DNS queries
# http.log — HTTP requests
# ssl.log  — TLS info
# files.log — file transfers

# Analyze Zeek conn.log for beaconing
cat conn.log | zeek-cut id.orig_h id.resp_h duration | \
  sort | uniq -c | sort -rn | head

# ── SURICATA RULES ────────────────────────────
# Detect reverse shell via nc
alert tcp any any -> any any (
  msg:"POSSIBLE NETCAT REVERSE SHELL";
  content:"bash -i"; nocase;
  sid:1000001; rev:1;
)

# Detect Mimikatz
alert smb any any -> $HOME_NET any (
  msg:"MIMIKATZ LSASS DUMP";
  content:"sekurlsa"; nocase;
  sid:1000002; rev:1;
)

# ── SNORT RULES ───────────────────────────────
# SQL injection detection
alert tcp any any -> $HTTP_SERVERS $HTTP_PORTS (
  msg:"SQL INJECTION ATTEMPT";
  pcre:"/(\%27)|(\')|(\-\-)|(%23)|(#)/i";
  classtype:web-application-attack;
  sid:2000001; rev:1;
)
Beaconing Detection
Malware often "beacons" home at regular intervals (every 60s, every 5min). Look for connections to the same external IP at suspiciously regular intervals. In Splunk: | stats count, range(_time) as duration, values(dest_port) by src_ip, dest_ip | eval beacon_interval = duration/count. Low variance = likely beaconing.

Linux live response triage
bash
# ── COLLECT VOLATILE DATA FIRST ───────────────
# Order matters! Most volatile → least volatile
date; hostname; uname -a        # system info
whoami; id; who; w               # current users
ps aux                          # running processes
ss -tulpn                       # network connections
netstat -tulpn                  # listening ports
ip addr; ip route                # network config
arp -a                          # ARP cache
cat /etc/hosts                   # hosts file tampering?
cat /etc/crontab                 # persistence via cron?
find / -perm -u=s -type f 2>/dev/null  # SUID files
last; lastlog                    # login history
cat ~/.bash_history              # command history

# ── COLLECT IOCs ──────────────────────────────
# File hashes (for threat intel lookup)
find / -mtime -1 -type f 2>/dev/null | \
  xargs sha256sum 2>/dev/null    # recently changed files

# Suspicious connections
ss -tn state established | grep -v "127.0.0.1\|::1"

# ── CONTAINMENT ───────────────────────────────
# Block IP at firewall
iptables -A INPUT -s 1.2.3.4 -j DROP
iptables -A OUTPUT -d 1.2.3.4 -j DROP

# Kill malicious process
kill -9 <PID>

# Isolate host (keep management access)
# Route all traffic to null except your IP
iptables -P INPUT DROP
iptables -P OUTPUT DROP
iptables -A INPUT -s YOUR_IP -j ACCEPT
iptables -A OUTPUT -d YOUR_IP -j ACCEPT

Threat Hunting

Proactively searching for hidden threats that have bypassed automated detection. Hypothesis-driven investigation using known attacker TTPs and behavioral analytics.

THREAT HUNTING PROCESS 1. HYPOTHESIS "Attackers using living-off-the-land binaries to avoid AV detection" "Attacker has established persistence via scheduled tasks" "There is lateral movement occurring via WMI" ↓ 2. DATA COLLECTION Which logs cover this hypothesis? (Process creation? Network? Auth?) ↓ 3. HUNT EXECUTION Write SIEM query / hunt in EDR / analyze raw logs ↓ 4. ANALYZE RESULTS No findings → Improve detection coverage Findings → Escalate to IR → Create detection rule ↓ 5. FEEDBACK LOOP Update detection rules · Document TTPs found · Share IOCs
Threat hunting queries — common hypotheses
SPL / bash
| ═══ HYPOTHESIS: Living-Off-The-Land (LOLBins) usage ═══════════════════
| Attackers use built-in tools to avoid detection
index=windows EventCode=4688
| where match(New_Process_Name,
  "(?i)certutil|mshta|regsvr32|wscript|cscript|bitsadmin|msiexec|rundll32|installutil|regasm|msbuild")
| table _time, host, Account_Name, New_Process_Name, Process_Command_Line
| where len(Process_Command_Line) > 200  | ← long cmdlines = encoded payloads

| ═══ HYPOTHESIS: PowerShell Encoded Commands ════════════════════════════
index=windows EventCode=4688 New_Process_Name="*powershell*"
| where match(Process_Command_Line, "(?i)encodedcommand|enc |hidden|bypass|noprofile|noninteractive")
| rex field=Process_Command_Line "(?i)-enc(?:odedcommand)?\s+(?P[A-Za-z0-9+/=]+)"
| eval decoded_payload=base64decode(b64_payload)
| table _time, host, Account_Name, decoded_payload

| ═══ HYPOTHESIS: Scheduled Task Persistence ════════════════════════════
index=windows EventCode=4698
| rex field=Message "Task Name:\s+(?P[^\n]+)"
| rex field=Message "Task Content:\s+(?P.+)"
| where match(task_content, "(?i)powershell|cmd|wscript|http|temp|appdata")
| table _time, host, Account_Name, task_name, task_content

| ═══ HYPOTHESIS: Kerberoasting ══════════════════════════════════════════
| Look for TGS requests for multiple service accounts
index=windows EventCode=4769 Failure_Code="0x0" Ticket_Encryption_Type="0x17"
| stats count, dc(Service_Name) as unique_services by Account_Name, Client_Address
| where unique_services > 3
| sort -unique_services

# ═══ HYPOTHESIS: Beaconing via bash ═════════════════════════════════════
# Look for regular outbound connections at fixed intervals
tshark -r capture.pcap -Y "tcp && ip.dst != 192.168.0.0/16" \
  -T fields -e ip.src -e ip.dst -e frame.time_relative | \
  awk '{print $2, $3}' | sort | uniq -c | sort -rn | head

EDR & Endpoint Defense

Endpoint Detection and Response platforms provide deep visibility into host activity, behavioral detection, and rapid response capabilities.

EDR Capabilities

  • Process tree visibility — see parent→child relationships
  • File system monitoring — what files are created/modified/deleted
  • Registry monitoring — persistence mechanism detection
  • Network connections — per-process outbound connections
  • Memory scanning — detect injected code, shellcode
  • Behavioral detection — pattern-based (no sig required)
  • Remote isolation — cut off infected host instantly
  • Response actions — kill process, delete file, collect memory

MITRE ATT&CK Mapping

The industry-standard framework for describing adversary behavior. Maps attacker Tactics → Techniques → Procedures to help defenders detect, respond, and improve coverage.

MITRE ATT&CK TACTICS — Enterprise (left→right = kill chain) TA0043 TA0042 TA0001 TA0002 TA0003 TA0004 TA0005 TA0006 TA0007 TA0008 TA0009 TA0011 TA0010 TA0040 Recon Resrc Initial Exec Persist PrivEsc Defense Cred Discov Lateral Collect C2 Exfil Impact Dev Access ment ation Evasion Access ery Move Each tactic contains multiple techniques, and each technique has sub-techniques Example: T1059 - Command and Scripting Interpreter → T1059.001 - PowerShell → T1059.003 - Windows Command Shell → T1059.004 - Unix Shell

Key ATT&CK Techniques — Detection & Response

ATT&CKDetection
TechniqueIDHow to DetectResponse
Spearphishing Attachment T1566.001 Email gateway alerts, suspicious attachment types, macro execution Block attachment, quarantine email, check for execution
PowerShell Execution T1059.001 Event 4688, ScriptBlock logging (4104), encoded commands Investigate cmdline, check parent process, block if malicious

Sigma Rules — ATT&CK Detection
Sigma is a generic YAML rule format for writing detection logic that converts to any SIEM (Splunk, ELK, QRadar). Example: sigmac -t splunk -r rules/windows/process_creation/proc_creation_win_powershell_encoded_cmd.yml. The Sigma repository has 2000+ community detection rules aligned to ATT&CK techniques.

  7. True positive → escalate + open ticket
